## 1. Имена и scope HTTP-cookies

**Вопрос:** Как назвать cookies и какой `Path` задать для `access_token` и `refresh_token`?

**Варианты:**

- [ ] A) `access_token` — `Path=/`, `refresh_token` — `Path=/api/session`
- [ ] B) Другие имена (например `__Host-access`, `__Host-refresh`)
- [ ] C) Свой вариант (описать ниже)

**Пояснение:**  
`Path=/api/session` для refresh ограничивает отправку cookie только эндпоинтами `/api/session/check`, `/api/session/refresh` и их потомками. Access при этом уходит на все запросы к API. Оба cookie — `HttpOnly`, `Secure`, домен из `SESSION_COOKIE_DOMAIN`.

**Рекомендация:** вариант A — простой и предсказуемый для SPA.

**Ответ:** access_token refresh_token. refresh_token только для check и refresh ручек.

## 2. Атрибут SameSite

**Вопрос:** Какое значение `SameSite` использовать для session-cookies?

**Варианты:**

- [ ] A) `Lax` — стандарт для same-site SPA (фронт и API на одном домене через Traefik)
- [ ] B) `Strict` — жёстче, может мешать некоторым сценариям перехода по ссылкам
- [ ] C) `None` + `Secure` — только если фронт и API на разных сайтах (cross-site)
- [ ] D) Свой вариант

**Пояснение:**  
При схеме `https://xxyyzz.localhost` (один origin для фронта и `/api`) обычно достаточно `Lax`. `Strict` усложняет UX при внешних редиректах. `None` нужен редко и требует явного CORS с credentials.

**Рекомендация:** `Lax` для dev и production при общем домене.

**Ответ:** Lax

## 3. Хранение токенов в Redis

**Вопрос:** Хранить `access_token` и `refresh_token` в Redis в открытом виде или только хеш?

**Варианты:**

- [ ] A) Хеш (HMAC-SHA256 или SHA-256) — в cookie plain, в Redis hash; lookup по hash
- [ ] B) Plain text в Redis — проще отладка, выше риск при утечке дампа Redis
- [ ] C) Свой вариант

**Пояснение:**  
Токены — 64 случайных символа, не JWT. При компрометации Redis plain-токены сразу дают доступ к сессиям. Хеширование — обычная практика для server-side session stores.

**Рекомендация:** вариант A.

**Ответ:** да, только хеши access и refresh токенов. Нужен будет стабильный алгоритм хеширования. Можно использовать object-hash

## 4. Поле `expiredAt` в ответе `/api/session/check`

**Вопрос:** Что именно возвращать в поле `expiredAt` (и нужны ли дополнительные поля)?

**Варианты:**

- [ ] A) Только истечение **access** (~10 мин) — фронт знает, когда снова вызывать check
- [ ] B) Только истечение **всей сессии** / refresh (~7 дней)
- [ ] C) Оба: `accessExpiresAt` + `sessionExpiresAt`
- [ ] D) Свой вариант

**Пояснение:**  
Фронтенду для polling/check обычно важен TTL access. Для UI «сессия активна до …» — TTL refresh. Одно поле `expiredAt` без уточнения создаёт путаницу.

**Рекомендация:** вариант C — явно и без двусмысленности.

**Ответ:** вариант C — явно и без двусмысленности.

## 5. Поведение при истёкшем refresh

**Вопрос:** Что делать, если refresh истёк или сессия не найдена (вызов `/check` или `/refresh`)?

**Варианты:**

- [ ] A) `401 Unauthorized` + `Set-Cookie` с удалением обоих cookies
- [ ] B) `401` без очистки cookies (фронт чистит сам)
- [ ] C) `403` + очистка cookies
- [ ] D) Свой вариант

**Пояснение:**  
Очистка cookies на сервере (`Max-Age=0`) гарантирует согласованное состояние даже если фронт не обработал ошибку.

**Рекомендация:** вариант A.

**Ответ:** `401 Unauthorized` + `Set-Cookie` с удалением обоих cookies

## 6. `REUSE_DETECTION_MODE=quarantine`

**Варианты:**

- [ ] A) **quarantine** — пометить сессию, отклонить запрос, опционально отозвать все сессии пользователя
- [ ] B) **reject** — только `401` для этого запроса, сессию не трогать
- [ ] C) **quarantine** только текущую `session_id`, остальные сессии пользователя живы
- [ ] D) **quarantine** — отозвать все сессии пользователя (logout everywhere)
- [ ] E) Свой вариант (опиши режимы)

**Пояснение:**  
В compose уже есть `REUSE_DETECTION_MODE=quarantine`. Нужно зафиксировать точную семантику до реализации rotate-логики в Redis.

**Рекомендация:** уточнить между C и D — отзыв одной сессии vs всех сессий пользователя.

**Ответ:** только `401` для этого запроса, сессию не трогать. Простое решение. По идеи истекшие токены даже хранить не будем.

## 9. Декораторы `@User()` и `@SessionUser()` без guard

**Вопрос:** Как разделить поведение параметрических декораторов?

**Варианты:**

- [ ] A) `@SessionUser()` → `PublicUser | null` без guard; `@User()` → всегда `PublicUser`, иначе `401` (guard не обязателен)
- [ ] B) `@SessionUser()` → optional; `@User()` работает только вместе с `AuthenticatedGuard`
- [ ] C) Один декоратор `@CurrentUser({ required: true | false })`
- [ ] D) Свой вариант

**Пояснение:**  
Глобальный middleware вешает ленивый `getUser()` на request. Вопрос — должен ли `@User()` сам бросать 401 или только guard.

**Рекомендация:** вариант A — middleware + `@SessionUser()` для публичных роутов с опциональной авторизацией; `@User()` для обязательной.

**Ответ:** `@User()` сам бросать 401 не должен. И давай будет только @SessionUser(). Для гарда название покороче: AuthGuard

## 10. Поведение `AuthenticatedGuard`

**Вопрос:** Как интерпретировать `AuthenticatedGuard` и `new AuthenticatedGuard({ strict: false })`?

**Варианты:**

- [ ] A) `strict: true` (по умолчанию) — 401, если user `null`; `strict: false` — пропускает, user через `@SessionUser()`
- [ ] B) Guard проверяет только наличие валидного access, без загрузки user из БД
- [ ] C) Свой вариант

**Пояснение:**  
При `strict: false` бессмысленно использовать `@User()` (он всё равно кинет 401) — только `@SessionUser()`.

**Рекомендация:** вариант A.

**Ответ:** `strict: true` (по умолчанию). Загрузка пользователя из БД происходит при первом же вызове getUser (напоминаю: которая ленивая и работает через замыкания для того чтобы повторно не лазила в БД)

**Рекомендация:** A на будущее + по умолчанию same-origin в dev.

**Ответ:** Не нужен CORS — фронт и API всегда same-origin через Traefik
